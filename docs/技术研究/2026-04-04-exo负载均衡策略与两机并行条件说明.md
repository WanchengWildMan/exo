# exo 负载均衡与两机并行：策略、内存流与你的 9B 问题

> 日期：2026-04-04
> 场景：9B 模型 + 17000 token prompt + 两台 16GB Mac

---

## 你最关心的问题，先回答

**Q: 当前实现能否让两台 16G 的机器分摊显存，而不是一台饿死？**

**A: 能，但不会自动发生。** 原因如下三点：

1. 放置算法默认"最少节点优先"—— 如果一台机器放得下模型权重，它会选单机。
2. 即便你强制两机放置，放置检查只看模型权重大小，不看 17000 token 的运行时峰值。
3. 两机 Pipeline 模式下，每台加载约一半的层，KV cache 和激活也各自只算自己那半，
   所以**确实能大幅降低每台的内存压力**。

**结论：你需要显式设置 `min_nodes=2` 才能触发两机分摊，默认不会。**

**两层负载均衡各自的职责：**

| 层     | 做什么                             | 策略                       |
|--------|------------------------------------|----------------------------|
| 放置层 | 决定一个模型实例放在哪些节点上     | 最少节点 + 内存打分         |
| 调度层 | 同模型有多个实例时，新请求发给谁   | 最短队列（任务数最少优先）  |

> **你的问题属于放置层。** 调度层只在"同模型多实例"时才生效。

## 三、Pipeline 并行的数据流（两机场景）

当放置到两个节点时，推理时数据这样流动：

```
 用户 Prompt (17000 tokens)
        │
        ▼
 ┌─────────────────────────────┐
 │  Node A: Layer 0 ~ 15       │
 │                             │
 │  1. Tokenize prompt         │
 │  2. Prefill: 逐步处理      │
 │     (EXO_PREFILL_STEP_SIZE, │
 │      默认64, 可设512)       │
 │  3. 产出中间激活值           │
 │     shape: [17000, hidden]  │
 │  4. 本地KV cache: 只存      │
 │     Layer 0~15 的 K,V       │
 │                             │
 │  内存占用:                   │
 │  · 权重 ≈ 2.5GB (半数层)   │
 │  · KV(L0-15): ~1.4GB       │
 │  · 激活: ~0.5GB            │
 │  · 系统开销: ~2-3GB        │
 │  合计: ~6-7GB              │
 └──────────────┬──────────────┘
                │
                │ libp2p Ring 传输
                │ 传中间激活值 ≈ 0.5GB
                │
                ▼
 ┌─────────────────────────────┐
 │  Node B: Layer 16 ~ 31      │
 │                             │
 │  1. 接收中间激活值           │
 │  2. 继续 Prefill 后半       │
 │  3. 产出 logits             │
 │  4. 本地KV cache: 只存      │
 │     Layer 16~31 的 K,V      │
 │                             │
 │  内存占用:                   │
 │  · 权重 ≈ 2.5GB (半数层)   │
 │  · KV(L16-31): ~1.4GB      │
 │  · 激活: ~0.5GB            │
 │  · 系统开销: ~2-3GB        │
 │  合计: ~6-7GB              │
 └──────────────┬──────────────┘
                │
                ▼
           Token 输出
           (回传给 Node A → API → 用户)
```

**关键：Pipeline 模式下，每台只需要承担约一半的权重和一半的 KV cache。**
所以两台 16GB 是**完全可以跑 9B + 17k tokens** 的——前提是放置层选了两机。

## 四、你的场景：9B + 17000 tokens + 2×16GB

### 单机放置（当前默认行为）的内存账

```
┌───────────────────────────────────────────────────┐
│              单机 16GB 内存分布                      │
│                                                   │
│  macOS 系统 + 常驻进程          ~3-4 GB           │
│  Python + exo 框架本身          ~0.5 GB           │
│  ─────────────────────────────────────────        │
│  模型权重 (9B 4-bit)            ~5.0 GB           │
│  KV cache (32层×17000 tokens)   ~2.7 GB           │
│  Prefill 激活值 (峰值)          ~0.5-1.0 GB       │
│  ─────────────────────────────────────────        │
│  总计需要:                      ~11.7-13.7 GB     │
│  可用:                          ~12-13 GB         │
│                                                   │
│  → 勉强能放，或者直接 OOM ← ★                     │
└───────────────────────────────────────────────────┘
```

放置检查只看权重 5GB < 可用 12GB → **通过**。
但实际运行时 KV cache + 激活让峰值逼近极限 → **OOM 或被系统杀进程**。

### 两机放置（需要 min_nodes=2）的内存账

```
┌──────────────────────────────┐  ┌──────────────────────────────┐
│      Node A (16GB)           │  │      Node B (16GB)           │
│                              │  │                              │
│  系统 + exo      ~3.5 GB    │  │  系统 + exo      ~3.5 GB    │
│  权重(16层)      ~2.5 GB    │  │  权重(16层)      ~2.5 GB    │
│  KV(16层×17k)    ~1.4 GB    │  │  KV(16层×17k)    ~1.4 GB    │
│  激活(峰值)      ~0.5 GB    │  │  激活(峰值)      ~0.5 GB    │
│  ─────────────────────       │  │  ─────────────────────       │
│  总计:           ~7.9 GB    │  │  总计:           ~7.9 GB    │
│  剩余:           ~8.1 GB ✓  │  │  剩余:           ~8.1 GB ✓  │
└──────────────────────────────┘  └──────────────────────────────┘
```

**两机放置后，每台还有 8GB 余量，完全没有压力。**

### "它是否一定会让两个 16G 来抗？"

**不会自动这样做。** 原因链：

```
默认 min_nodes=1
       │
       ▼
单节点 cycle [A] 通过内存检查 (5GB权重 < 12GB可用)
       │
       ▼
get_smallest_cycles() 选 [A] 而非 [A,B]  ← 因为 1 < 2
       │
       ▼
模型放在 A 单机上运行
       │
       ▼
运行时 OOM (权重+KV+激活 ≈ 12-14GB > 可用 12-13GB)
```

## 五、解决方案：让两机自动生效

### 方案 1: API 自动创建路径（已有实现）

`api/main.py` L954 的 `_create_instance_for_model_on_demand` 函数
**已经**从 `node_count` 开始向下尝试：

```python
for min_nodes in range(node_count, 0, -1):
    # 先尝试 min_nodes=2，再尝试 min_nodes=1
```

所以通过 API 自动创建时，它**会先尝试两机方案**。
但如果两机方案的 cycle 没有通过内存检查（比如节点未就绪），
它会降级到单机。

### 方案 2: 手动创建实例时指定 min_nodes

```bash
# 在请求中显式设置
curl http://localhost:52415/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "your-9b-model", "messages": [...]}'
```

API 自动创建路径已经处理了这个。如果还是单机，说明可能只有一个节点
在集群里（网络发现问题）。

### 方案 3: 环境变量调优

```bash
# 所有节点都需要设置
# EXO_USE_TOTAL_MEMORY_FOR_PLACEMENT 已默认开启（=1），无需手动设置
# 若需回退到旧行为（用可用内存），设为 0：
# export EXO_USE_TOTAL_MEMORY_FOR_PLACEMENT=0

export EXO_KV_CACHE_BITS=2        # 量化 KV cache，内存降约 8 倍

# 增大 prefill 步长，减少峰值激活次数（但不减少总量）
export EXO_PREFILL_STEP_SIZE=512
```

### 方案 4: 跳过内存检查（开发调试用）

```bash
export EXO_SKIP_PLACEMENT_MEMORY_CHECK=1
```

**警告：** 这会跳过所有内存验证，可能在运行时 OOM。

## 七、一句话总结

> 放置算法以"单机能放就单机"为策略。你的 9B 权重只有 5GB，单台 16GB 看起来放得下，
> 但 17000 tokens 的 KV cache 额外要 2.7GB，实际跑不动。
> 通过 API 自动创建路径它会先尝试多机，但如果集群只报告了一个节点就会降级。
> **确认两台都在集群中可见**，API 自动创建路径就会优先尝试 `min_nodes=2`。
