# oMLX 内存管理分析与 exo 融合方案

**日期**: 2025-07-14  
**背景**: 两台 16GB Mac Mini (M2+M4) 运行 Qwen3.5-9B-MLX-4bit 时 OOM，需要研究 oMLX 的内存管理方案

## 1. oMLX 核心内存架构

### 1.1 三层缓存体系

```
┌──────────────────────────────────────────────────────────┐
│                   GPU 推理内存                            │
│  BatchGenerator 管理活跃请求的 KV Cache                   │
│  (与 mlx-lm 共享，不由 oMLX 直接管理)                     │
├──────────────────────────────────────────────────────────┤
│                  Hot Cache (RAM)                          │
│  内存中的 raw bytes 缓冲，write-back 模式                 │
│  LRU 淘汰时写入 SSD，读取时零 I/O                        │
│  可配置大小，默认关闭 (hot_cache_max_size=0)              │
├──────────────────────────────────────────────────────────┤
│                Cold Cache (SSD)                           │
│  safetensors 格式持久化，按 block hash 存储               │
│  后台写入线程 (纯 Python I/O，无 Metal API)               │
│  跨重启可复用，LRU 淘汰，磁盘容量感知                    │
└──────────────────────────────────────────────────────────┘
```

### 1.2 Block-based Paged Cache (vLLM 架构)

- **CacheBlock**: 每块 256 tokens (可配置)，ref_count 引用计数，双向链表 LRU
- **FreeKVCacheBlockQueue**: O(1) 分配/回收，fake head/tail 哨兵节点
- **BlockHashToBlockMap**: SHA256 chain hash 实现前缀去重
  - `hash = SHA256(parent_hash + token_ids + model_name)`
- **BlockTable**: 每请求的 logical → physical 块映射
- **动态块池**: 从 initial_blocks(256) 按需增长到 max_blocks
- **Copy-on-Write**: 共享块的 fork 操作

### 1.3 内存监控 (MemoryMonitor)

```python
# 使用 MLX Metal API 而非 psutil

peak = mx.metal.get_max_working_set_bytes()  # 峰值

# Baseline 追踪: 模型加载后捕获基线，KV 增长 = active - baseline

#   head_dim > 128: SDPA fallback → 完整注意力矩阵 [B,n_q,chunk,kv_len] float32
#   head_dim <= 128: fused kernel → O(n) 内存
```

### 1.4 进程级内存执行器 (ProcessMemoryEnforcer)

- 后台 asyncio 任务，每 1s 轮询 `mx.get_active_memory()`
- 默认上限: system_ram - 8GB (自适应: 20% 保留，clamp [2GB, 8GB])
- Hard limit: system_ram - 4GB (prefill 预检)
- **关键发现**: `mx.set_memory_limit()` 会导致 alloc/free 抖动推入 swap → 已禁用!
- 三种场景: 多模型(淘汰 LRU), 单模型(中止请求), 均在推理(先淘汰后中止)

## 2. exo 现有架构

### 2.1 KVPrefixCache (整体级缓存)

```
┌─────────────────────────────────────────┐
│         KVPrefixCache (LRU)             │
│  key: tuple(prompt_tokens)              │
│  value: (mlx_cache_list, tokens)        │
│  淘汰触发: psutil RAM > threshold       │
│  threshold: 自适应 (0.70 ~ 0.85)       │
└─────────────────────────────────────────┘
```

- **整体级**: 缓存完整的 prompt + KV cache 对，无块级拆分
- **前缀匹配**: `get_prefix_length()` 找最长公共前缀
- **SSM 快照**: `CacheSnapshot` 在已知 token 位置捕获 GatedDeltaNet 状态
- **RotatingKVCache**: max_size 限制 (EXO_MAX_KV_SIZE=3200)

### 2.2 Prefill 内存检查

```python
# 使用 psutil 检查系统 RAM (非 Metal 内存)
mem = psutil.virtual_memory()
avail_gb = mem.available / (1024 ** 3)
if avail_gb < EXO_PREFILL_MIN_AVAILABLE_GB:  # 默认 0.5GB
    raise PrefillOOM(...)

# 周期性 flush 计算图
if step % EXO_PREFILL_EVAL_INTERVAL == 0:  # 默认 8
    mx.eval([c.state for c in cache])
```

### 2.3 Cleanup 流程

```python
# pre-prefill:
mx.clear_cache()  # ⚠️ 无 mx.synchronize() 保护!
gc.collect()
kv_prefix_cache._evict_if_needed()

# post-task (所有任务完成时):
mx.clear_cache()  # ⚠️ 同上
gc.collect()
```

## 3. 关键差异对比

| 维度 | oMLX | exo |
|------|------|-----|
| **内存监控** | `mx.get_active_memory()` (Metal GPU) | `psutil.virtual_memory()` (系统 RAM) |
| **缓存粒度** | 块级 (256 tokens/block) | 整体级 (完整 prompt) |
| **前缀复用** | SHA256 chain hash + block 级去重 | 线性扫描最长公共前缀 |
| **SSD 卸载** | ✅ safetensors + 后台写入 | ❌ 无 |
| **Hot 缓存** | ✅ RAM 内 raw bytes 缓冲 | ❌ 无 |
| **LRU 淘汰** | O(1) 双向链表 | O(n) 遍历 |
| **内存限制** | 绝对字节 (RAM - 8GB) | 百分比阈值 (0.70-0.85) |
| **clear_cache** | `mx.synchronize()` + `mx.clear_cache()` + 延迟 8 步 | 直接 `mx.clear_cache()` |
| **Prefill 预检** | 峰值内存估算 (SDPA head_dim 感知) | 仅检查可用 RAM |
| **set_memory_limit** | 已禁用 (导致 swap 抖动) | 未使用 |
| **ArraysCache** | 支持混合模型 (boundary snapshot) | 保留 ArraysCache 不替换 |
| **分布式** | 单机多模型 | 多机流水线并行 |

## 4. 可融合组件评估

### 4.1 P0 — 立即可集成 (低工作量，高收益)

#### A. `mx.synchronize()` 保护 `mx.clear_cache()` ⭐⭐⭐

**问题**: oMLX 发现不同步直接 clear 可导致 M4 内核崩溃 (#300, #435)
```python
# oMLX 的做法:
def _sync_and_clear_cache():
    mx.synchronize(generation_stream)
    mx.synchronize()  # default stream
    mx.clear_cache()
```

**exo 现状**: `mx.clear_cache()` 在 batch_generate.py 中直接调用，无 sync  
**风险**: M4 上可能触发 'completeMemory() prepare count underflow' 内核 panic  
**改动**: ~5 行代码

#### B. 使用 `mx.get_active_memory()` 替代 psutil ⭐⭐⭐

**问题**: `psutil.virtual_memory()` 反映系统 RAM 使用，包含 OS 缓存、其他进程  
**oMLX 做法**: 直接用 `mx.get_active_memory()` 追踪 Metal 实际分配  
**优势**: 更精确反映 MLX 推理的实际内存压力  
**改动**: ~10 行代码 (generate.py 中的 `_check_memory_during_prefill`)

#### C. Prefill 前峰值内存估算 ⭐⭐

**问题**: exo 仅在 prefill 过程中检查内存，事后告警  
**oMLX 做法**: prefill 前估算峰值 = KV cache + SDPA attention matrix
```
head_dim > 128: peak = chunk_size × kv_len × n_heads × 4 bytes (float32 全矩阵)
head_dim <= 128: peak ≈ tiled output buffer only (较小)
```
**优势**: 在 prefill 开始前就知道会不会 OOM  
**改动**: ~30 行代码

### 4.2 P1 — 中期可集成 (中工作量，高收益)

#### D. 延迟清理 Metal 缓存 ⭐⭐

**问题**: 请求完成后立即 `mx.clear_cache()` 与 IOKit 异步回调竞争  
**oMLX 做法**: 延迟 8 个生成步骤后再清理  
**改动**: ~20 行代码

#### E. 进程级内存执行器 ⭐⭐

**oMLX 做法**: 后台 asyncio 任务每秒轮询 Metal 内存，超限时淘汰/中止  
**对 exo 价值**: 当前 exo 仅在 prefill 时检查，generation 阶段无保护  
**改动**: ~100 行新代码

### 4.3 P2 — 长期可探索 (高工作量，颠覆性收益)

#### F. Block-based SSD KV Cache 卸载 ⭐⭐⭐⭐

这是用户最初询问的 "token存盘" 功能。

**挑战**:
1. exo 使用流水线并行 (M4=layers 0-15, M2=layers 16-31)，每节点只有半层
2. oMLX 的 BlockPool 假设单机所有层
3. `RotatingKVCache` 的旋转窗口需要对齐块边界
4. `ArraysCache` (GatedDeltaNet) 的 boundary snapshot 需要特殊处理

**可行路径**:
- 方案 A: 每节点独立的 SSD cache (只缓存本节点负责的层)
- 方案 B: 仅缓存 KV 层，GatedDeltaNet 层每次重算 (因为 ArraysCache 是 O(1) 的)
- 方案 C: 引入 oMLX 的 TieredCacheManager 但仅用 SSD-only 模式

**预估改动**: 500-1000 行代码，需要重构 KVPrefixCache

```

第三阶段 (长期):
  └─ F. Block-based SSD KV Cache 卸载           (~500-1000 行)
```

## 6. oMLX 的重要教训 (避坑指南)

1. **`mx.set_memory_limit()` 是个坑** — 导致 alloc/free 抖动，推系统入 swap (#429)
2. **`mx.clear_cache()` 前必须 `mx.synchronize()`** — 否则 M4 内核 panic (#300)
3. **延迟清理** — 请求完成后不要立即清理，等 IOKit 回调完成 (#435)
4. **bfloat16 存盘** — 需要 `view(uint16)` trick，safetensors 不直接支持 bf16
5. **后台写入线程不能调 MLX API** — Metal 不是线程安全的，tensor bytes 必须在推理线程提取
6. **Prefill 峰值取决于 head_dim** — >128 时 SDPA fallback 会分配完整注意力矩阵 (float32)
7. **psutil 不如 mx.get_active_memory()** — 前者包含 OS 缓存等噪声

## 7. Qwen3.5-9B 特殊考虑

Qwen3.5 是混合模型: 标准 Attention (KVCache) + GatedDeltaNet (ArraysCache)

- oMLX 对此有专门支持: `CacheList` 嵌套多种 cache type
- `boundary_snapshot` 在 block 边界捕获 ArraysCache 状态
- `_ARRAYS_CACHE_BLOCK_SIZE = 2048` — 更大块减少 snapshot 开销
- exo 已正确识别这一点: `make_kv_cache()` 保留 ArraysCache 不替换

对于 SSD 卸载:
- KV 层: 可以安全卸载/恢复 (标准 key/value tensor)
- GatedDeltaNet 层: 需要 boundary snapshot (递归状态，不能任意切割)
- 最简方案: **只卸载 KV 层，GatedDeltaNet 每次重算** (因为 O(1) 开销)
