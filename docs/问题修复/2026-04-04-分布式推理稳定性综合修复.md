## 1. Placement 默认使用总内存

### 现象

两台 16GB Mac Mini 启动后，placement 算法因系统占用导致 `ram_available` 波动
（通常仅 9-11 GB），计算可用内存不稳定，频繁出现 placement 失败。

### 根因

`EXO_USE_TOTAL_MEMORY_FOR_PLACEMENT` 默认关闭，placement 使用 `ram_available`。
16GB 机器上，操作系统+后台进程消耗约 5-7 GB，导致可用内存虚低。
实际推理时 macOS 会释放文件缓存，真实可用实际接近 `ram_total`。

### 修复

将 `EXO_USE_TOTAL_MEMORY_FOR_PLACEMENT` 默认值从 `"0"` 改为 `"1"`。

**文件 1：`src/exo/master/placement_utils.py`（L23-28）**

```python
def _use_total_memory_for_placement() -> bool:
    return os.getenv("EXO_USE_TOTAL_MEMORY_FOR_PLACEMENT", "1").lower() not in {
        "0", "false", "no", "off",
    }
```

**文件 2：`src/exo/master/placement.py`（L115-117）**

```python
use_total_memory_for_placement = os.getenv(
    "EXO_USE_TOTAL_MEMORY_FOR_PLACEMENT", "1"
).lower() not in {"0", "false", "no", "off"}
```

### 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `EXO_USE_TOTAL_MEMORY_FOR_PLACEMENT` | `1` | 使用 ram_total 代替 ram_available |

## 2. API 内存门 bypass 修复

### 现象

即使设置了 `EXO_USE_TOTAL_MEMORY_FOR_PLACEMENT=1`，API 层的 auto-create 仍因
"内存不足"拒绝创建实例。

### 根因

`src/exo/api/main.py` 中的 `_calculate_total_available_memory()` 独立于
placement 算法，始终使用 `memory.ram_available`，未读取同一个环境变量。
导致 placement 用总内存通过了，但 API 层用可用内存又拒绝了。

### 修复

**文件：`src/exo/api/main.py`（`_calculate_total_available_memory()`）**

```python
def _calculate_total_available_memory(self) -> Memory:
    use_total = os.getenv(
        "EXO_USE_TOTAL_MEMORY_FOR_PLACEMENT", "1"
    ).lower() not in {"0", "false", "no", "off"}
    total_available = Memory()
    for memory in self.state.node_memory.values():
        total_available += memory.ram_total if use_total else memory.ram_available
    return total_available
```

现在 API 层与 placement 算法使用一致的内存口径。

## 3. Prefill OOM 保护

### 现象

长 prompt（如 9916 token）prefill 时，R0 因内存耗尽被 macOS 杀掉。
R1 则在 `mx.distributed.all_sum` 上无限阻塞——MLX 的集体通信没有超时机制。

### 根因

Prefill 按 chunk 逐步处理 prompt，每个 chunk 分配 KV cache。
16GB 机器上，模型权重 ~2.5GB + 系统开销后，可用于 KV 的空间有限。
长 prompt 处理到中途内存耗尽，进程被 OOM killer 终结。

### 修复

新增主动内存监控，在接近阈值时提前中止 prefill。

**文件：`src/exo/worker/engines/mlx/generator/generate.py`**

#### 3a. PrefillOOM 异常（L116）

```python
class PrefillOOM(BaseException):
    """Raised when memory usage exceeds threshold during prefill."""
```

#### 3b. 内存检查函数（L124-145）

```python
def _check_memory_during_prefill(step: int, processed: int, total: int) -> None:
    if step % _PREFILL_MEMORY_CHECK_INTERVAL != 0:
        return
    mem = psutil.virtual_memory()
    used_pct = mem.percent / 100.0
    if used_pct > _PREFILL_MEMORY_THRESHOLD:
        raise PrefillOOM(
            f"Memory usage {used_pct:.1%} exceeded threshold "
            f"{_PREFILL_MEMORY_THRESHOLD:.1%} at prefill token {processed}/{total}"
        )
```

每 N 步（默认 8 步）检查一次系统 RAM 使用率，超过阈值时抛出 `PrefillOOM`。

#### 3c. 异常处理

**文件：`src/exo/worker/runner/llm_inference/batch_generator.py`**

`SingleGenerator.step()` 和 `BatchGenerator` 的任务提交逻辑均捕获 `PrefillOOM`，
将其转化为错误消息返回给客户端，而非让进程崩溃。

```python
except (PrefillOOM, PrefillSyncTimeout) as e:
    logger.error(f"Prefill failed for task {task.task_id}: {e}")
    self._send_error(task, RuntimeError(str(e)))
```

### 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `EXO_PREFILL_MEMORY_CHECK_INTERVAL` | `8` | 每 N 步检查一次内存 |
| `EXO_PREFILL_MEMORY_THRESHOLD` | `0.92` | RAM 使用率阈值（92%） |

### 注意

当前配置 `start-my.sh` 设置 `EXO_PREFILL_MEMORY_THRESHOLD=0.95`，
在 95% RAM 使用率时主动中止 prefill。实测发现禁用（=1）后，macOS 会在
~95% 时以 signal 6 (Abort) 杀掉 runner 进程，导致分布式 ring 另一端
永久等待。0.95 的阈值可在被杀前主动优雅退出。

## 4. 分布式 Sync 超时保护

### 现象

Pipeline 并行推理中，当一个 rank（如 R0）OOM 崩溃后，另一个 rank（R1）
在 `mx.distributed.all_sum` / `all_gather` 上**无限等待**。
MLX 的集体通信是阻塞操作，没有原生超时参数。

表现为 R1 日志停在：
```
[Prefill memory] step=0 tokens=0/310 ram_used=48.5% ram_available=7.72GB
```
此后无任何输出，进程完全卡死。

### 根因

MLX distributed 的 `all_sum`/`all_gather` 底层使用 MPI 或自定义通信，
当对端进程退出后，本端调用永远不会返回也不会抛出异常。

### 修复

用 `concurrent.futures.ThreadPoolExecutor` 包装同步回调，加上超时。

**文件：`src/exo/worker/engines/mlx/generator/generate.py`**

#### 4a. PrefillSyncTimeout 异常（L147）

```python
class PrefillSyncTimeout(BaseException):
    """Raised when a distributed sync during prefill exceeds the timeout."""
```

#### 4b. 超时包装函数（L154-185）

```python
def _run_prefill_sync(
    callback: Callable[[], None] | None,
    *, rank: int, phase: str,
) -> None:
    if callback is None:
        return
    if _PREFILL_SYNC_TIMEOUT <= 0:
        callback()
    else:
        with ThreadPoolExecutor(max_workers=1) as pool:
            future = pool.submit(callback)
            try:
                future.result(timeout=_PREFILL_SYNC_TIMEOUT)
            except FuturesTimeoutError:
                raise PrefillSyncTimeout(
                    f"[R{rank}] sync timed out after {elapsed}s during {phase}"
                ) from None
```

Pipeline prefill 的所有 distributed callback（leading dummies、real chunks、
trailing dummies）均通过此函数调用，每个阶段独立计时。

#### 4c. 异常处理

与 `PrefillOOM` 共用同一套处理逻辑（见第 3 节 3c）。

### 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `EXO_PREFILL_SYNC_TIMEOUT` | `60` | 单次 sync 超时（秒），0=禁用 |

## 5. 系统提示裁剪

### 现象

OpenClaw 发送的系统提示包含完整的 skills 列表（~40 个技能的描述、路径），
单个系统提示长达 15000+ 字符（约 5000+ token）。
这意味着 9B 模型的有效上下文被大幅压缩，且 prefill 时间显著增加。

### 修复

在 `chat_request_to_text_generation()` 解析消息时，对 system 消息执行三级裁剪。

**文件：`src/exo/api/adapters/chat_completions.py`**

```python
def _trim_system_prompt(text: str) -> str:
    # 1. 若 EXO_STRIP_SKILLS_BLOCK 开启：完整移除 skills 块
    if _STRIP_SKILLS_BLOCK:
        text = _SKILLS_BLOCK_RE.sub("", text).strip()
    # 2. 否则若 EXO_SKILLS_DESC_MAX_CHARS 有值：截断每个 skill 描述
    elif _SKILLS_DESC_MAX_CHARS is not None:
        text = _SKILLS_BLOCK_RE.sub(
            lambda m: _trim_skill_descriptions(m.group(1), _SKILLS_DESC_MAX_CHARS),
            text,
        )
    # 3. 最后若 EXO_MAX_SYSTEM_PROMPT_CHARS 有值且超长：截断整体
    if _MAX_SYSTEM_PROMPT_CHARS is not None and len(text) > _MAX_SYSTEM_PROMPT_CHARS:
        text = text[: _MAX_SYSTEM_PROMPT_CHARS] + " [trimmed]"
    return text
```

同时在处理入口添加了日志，记录裁剪前后的字符数。

### 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `EXO_STRIP_SKILLS_BLOCK` | 空（禁用） | 完整移除 skills XML 块 |
| `EXO_SKILLS_DESC_MAX_CHARS` | 空（禁用） | 每个 skill 描述最大字符数 |
| `EXO_MAX_SYSTEM_PROMPT_CHARS` | 空（禁用） | 系统提示总字符数上限 |

### 当前配置

`start-my.sh` 设置：
```bash
EXO_SKILLS_DESC_MAX_CHARS=30
EXO_MAX_SYSTEM_PROMPT_CHARS=3000
```

效果：原始 ~15000 字符 → 裁剪后 ~3000 字符，prefill token 数减少约 60%。
