# Decode 阶段内存安全阀 (P1)

日期: 2026-04-17
前置: P0 decode sync+clear 间隔从 512→1（同日完成）

## 问题

即使有了 P0 的每步 sync+clear，在以下场景中仍然可能 OOM：
- KV cache 随生成 token 数增加而持续增长
- 当 KV cache + 模型权重 + OS 已经占满大部分内存时
- 下一步 forward pass 的瞬时峰值会超出限制

P0 解决的是"中间张量累积"问题，P1 解决的是"KV cache 增长到极限"问题。

## 解决方案

### 原理（对齐 oMLX ProcessMemoryEnforcer）

在 decode 的**每一步开始**（forward pass 之前）检查 Metal 活跃内存。
如果已经超过阈值，**跳过 forward pass**，直接返回 `finish_reason="length"`。

这比 OOM crash 好得多——用户得到一个截断但完整的响应。

### 实现

**环境变量**: `EXO_DECODE_MAX_METAL_GB`
- 默认 `0` = 自动计算（`total_ram - 1.5GB`）
- 与 prefill 的 `EXO_PREFILL_MAX_METAL_GB` 相同公式
- 区别：prefill 超限时 raise PrefillOOM；decode 超限时 graceful finish

**Batch 路径** (`opt_batch_gen.py` → `_fast_next()`):
- 在 `self.model(prev_tokens[:, None], cache=batch.cache)` 之前检查
- 超限时：构造所有 entry 的 Response（使用上一步的 token），finish_reason="length"
- 设置 `self.active_batch = None` 停止 batch
- 不运行 forward pass，不采样——完全避免 OOM

**Stream 路径** (`generate.py` → `mlx_generate()` 循环):
- 在每步 sync+clear 之后检查 Metal 活跃内存
- 超限时：设置 `_decode_memory_exceeded = True`
- 在 finish_reason 判断处：`if _decode_memory_exceeded: finish_reason = "length"`
- 当前 token 正常 yield，之后 `is_done = True` 终止循环

### 涉及文件

| 文件 | 修改 |
|------|------|
| `src/exo/worker/engines/mlx/patches/opt_batch_gen.py` | 新增 `_get_decode_memory_limit_bytes()`，`_fast_next()` 入口安全阀 |

| `scripts/start-my.sh` | 新增 `EXO_DECODE_MAX_METAL_GB=0` |

### 内存限制计算

```
EXO_DECODE_MAX_METAL_GB > 0 时:
    limit = EXO_DECODE_MAX_METAL_GB (GB)

EXO_DECODE_MAX_METAL_GB = 0 (默认) 时:
    limit = total_ram - 1.5 GB
    例: 16GB 机器 → limit = 14.5 GB
```

### 与 P0 的配合

```
Decode step N:
  1. mx.synchronize() + mx.clear_cache()  ← P0: 每步清理中间张量
  2. 检查 mx.get_active_memory()           ← P1: 安全阀
     如果超限 → finish_reason="length", 跳过 forward pass
  3. model(prev_tokens, cache=...)          ← 只有在限额内才执行
```

## 与 oMLX 的差距

已对齐:
- ✅ P0: decode 阶段每步 sync+clear（oMLX 也是每步清理）
- ✅ P1: decode 内存安全阀（对标 oMLX ProcessMemoryEnforcer 的被动保护）

未对齐:
- ❌ 进程级内存预算（oMLX: `--max-process-memory`，主动限制而非被动检查）
- ❌ 块级 KV cache + 逐块 SSD 驱逐（oMLX: paged_cache + paged_ssd_cache）
- ❌ 持续内存监控线程（oMLX: memory_monitor.py）
