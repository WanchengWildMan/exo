# 2026-04-17 分布式 Prefill 同步失败修复

## 问题现象

双机分布式推理 (M4 R0 + M2 R1) 在 prefill 阶段互相超时：
- R1 在 `leading_dummy 1/1` 等待 R0 60秒后超时
- R0 在 `real_chunk 1/9` 看似成功（5.1ms），但 `real_chunk 2/9` 又超时

## 根因分析（两层问题）

### 第一层：R0 的 model() 阻塞了 ~9 分钟

完整日志时间线：

```
R0:
16:52:34.6451  KV cache hit: 16572/17145 tokens cached (96.7%)
16:52:34.6545  Starting prefill         ← mx_barrier 通过，两边同时开始
16:52:34.6551  [R0] Pipeline prefill: 9 real + 0 leading + 1 trailing
16:52:34.6554  [Prefill memory] metal_active=3.21GB
               ← 然后进入 model() ... 沉默 9 分钟 ...
16:58:17.6880  Worker plan: CancelTask  ← worker 线程收到取消（但 runner 阻塞中）
17:01:26.7179  [R0] Waiting for sync: real_chunk 1/9  ← model() 终于返回

R1:
16:52:34.6440  Starting prefill         ← 同时开始！mx_barrier 工作正常
16:52:34.6443  [R1] Pipeline prefill: 9 real + 1 leading + 0 trailing
16:52:34.6446  [R1] Waiting for sync: leading_dummy 1/1  ← 2ms 后到达 sync（无 model 调用）
16:52:34.6448  [R1] checking cancellations  ← 进入 MPI 集合操作
16:53:34.6552  [R1] timed out after 60.0s   ← R0 还没到，超时！
```

**两边同时通过 mx_barrier，但 R0 的 `model()` 调用（包含 `PipelineLastLayer.mx.eval(output)`）
阻塞了约 9 分钟。** R1 此时在 leading_dummy sync 等待 R0 的匹配 MPI 集合操作。

R0 `model()` 慢的原因待确认（已加诊断日志），可能是：
- `PipelineLastLayer.__call__()` 中的 `mx.eval(output)` 强制求值了整个计算图
- 首次运行特定 input shape 的 Metal kernel JIT 编译
- KV prefix cache 恢复后的首次 attention 计算（16572 token cache）
- Metal 内存分配抖动

### 第二层：超时后僵尸线程毒化 MPI 集合操作（Bug）

`_run_prefill_sync()` 使用 `ThreadPoolExecutor` 实现超时：
```python
with ThreadPoolExecutor(max_workers=1) as pool:
    future = pool.submit(callback)
    future.result(timeout=60)  # 超时抛异常
```

问题：**超时后 callback 线程（僵尸线程）仍然活着**，阻塞在 MPI 集合操作内部。

僵尸线程造成虚假同步：
1. T+0s: R1 callback 线程调用 `agree_on_cancellations()` → `all_sum` 阻塞
2. T+60s: R1 主线程超时 → `PrefillSyncTimeout` → R1 prefill 失败
3. T+60s: **僵尸线程仍在 `all_sum` 内等待**
4. T+540s: R0 model() 返回 → 调用 `agree_on_cancellations()` → **唤醒僵尸线程**
5. T+540s: 僵尸完成 3 个集合操作 → R0 认为 `real_chunk 1/9` 成功（5.1ms）
6. T+540s: 僵尸结束 → **不再有匹配的集合操作**
7. T+540s: R0 进入 `real_chunk 2/9` → 无人响应 → 阻塞 60s → 超时

核心本质：MPI 集合操作要求**所有 rank 按相同顺序调用**。
僵尸线程消耗了一组集合操作后死亡，导致后续操作永久不对齐。

## 修复方案

### 1. 移除 ThreadPoolExecutor 超时，改为直接阻塞

```python
def _run_prefill_sync(callback, *, rank, phase):
    # 直接调用 callback（阻塞），不使用线程超时
    callback()
```

依赖以下机制处理异常情况：
- `mx_barrier()` 在 prefill 前同步两个 rank
- libp2p 断连检测处理节点崩溃
- master 任务取消作为外部超时控制

### 2. 心跳日志

新增 `_heartbeat_watcher` 守护线程，每 30s 输出等待日志。

### 3. 诊断日志（定位 model() 慢的根因）

- `pipeline_parallel_prefill`: model() 调用前后计时
- `PipelineLastLayer.__call__()`: original_layer 和 mx.eval 分别计时
- `PipelineFirstLayer.__call__()`: recv 和 eval 分别计时
