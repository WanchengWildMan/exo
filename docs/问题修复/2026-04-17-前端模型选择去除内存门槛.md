# 2026-04-17 前端模型选择去除内存门槛

## 需求重述

用户反馈：API 已经可以成功请求 `mlx-community/Qwen3.5-27B-Claude-4.6-Opus-Distilled-MLX-4bit`，
但前端仍然无法选择该模型。

本次目标不是放开后端安全限制，而是只去掉前端基于“当前内存是否 fit”做的阻止选择逻辑：

1. 前端模型列表必须能显示并允许点击 27B。
2. 后端 placement / auto-create / 实例创建校验继续保留。
3. 前端仍可继续展示 `fits_now / fits_cluster_capacity / too_large` 状态，仅不再把它当作禁用条件。

## 现状分析

已确认：

1. `GET /v1/models` 可以返回该 27B 模型。
2. 直接调用 `/v1/chat/completions` 请求该模型已经 `HTTP 200` 成功。
3. 限制发生在 dashboard 前端，而不是 exo API。

具体限制点：

1. `dashboard/src/routes/+page.svelte`
   - `onboardingModels` 只保留 `hasEnoughMemory(model)` 为真的模型。
   - 传给 `ModelPickerModal` 的 `canModelFit` 回调直接使用 `hasEnoughMemory(model)`。
2. `dashboard/src/lib/components/ChatModelSelector.svelte`
   - 推荐模型选择逻辑只从 `fits in memory` 的模型里挑选。
3. `dashboard/src/lib/components/ModelPickerGroup.svelte`
   - 组件本身会根据 `canModelFit` 决定是否允许点击。

因此，真正需要修改的是父组件传入的“可选条件”和推荐列表过滤条件，而不是后端模型列表接口。

## 方案

只做前端去门槛：

1. 模型可选条件从“当前内存 fit”改为“模型有合法大小信息”。
2. 推荐列表不再按当前可用内存裁掉模型。
3. 继续保留 `getModelFitStatus()`，用于前端展示提醒与排序。
4. 后端不改，真实能否运行仍由 placement / 启动实例流程决定。

这样做的理由：

1. 当前前端内存估算是静态启发式，不等于真实 placement 结果。
2. 27B 已被真实 API 验证可以运行，继续让前端拦截就是错误限制。
3. 把选择权交还给用户，比在前端做武断阻断更合理。

## 修改结果

已完成以下改动：

1. `dashboard/src/routes/+page.svelte`
   - `onboardingModels` 不再按 `hasEnoughMemory(model)` 过滤。
   - 两处 `ModelPickerModal` 的 `canModelFit` 改为“只要模型大小信息有效即可选择”。
2. `dashboard/src/lib/components/ChatModelSelector.svelte`
   - 推荐模型选择逻辑从“必须 fits in memory”改为“模型可选即可”。
   - 因此前端推荐/选择层面不会再屏蔽 27B。

## 结果口径

修改后前端行为是：

1. 模型是否显示、是否可点：不再由前端内存估算决定。
2. 模型 fit 状态标签：仍保留，用于提示和排序。
3. 模型是否真的能启动：仍由后端 placement 和实例创建流程决定。

这保证了“前端不误杀，后端仍兜底”的职责边界。
